# Requirements Specification: REQ-GH-216

Make ATDD the default mode for all workflows — remove the `--atdd` and `--atdd-ready` CLI flags; replace flag-scoped conditionals with a config-driven behavior surface in `.isdlc/config.json`.

## Functional Requirements

- **FR-001** Remove the `--atdd` CLI flag from the build and test-generate workflow definitions in `src/isdlc/config/workflows.json`.
  - **Confidence**: High
  - AC-001-01: Given the build workflow is invoked without any ATDD-related flag, When Phase 05 begins, Then ATDD scaffolds are generated by default per the `atdd` config section.
  - AC-001-02: Given the legacy `--atdd` flag is passed to `/isdlc build`, When the CLI parses arguments, Then no special handling is required (the flag no longer exists; the unreleased framework does not warrant a deprecation layer).

- **FR-002** Remove the `--atdd-ready` CLI flag from the discover command.
  - **Confidence**: High
  - AC-002-01: Given `/discover` is invoked without `--atdd-ready` on an existing project, When the orchestrator reaches sub-phase 1d, Then the atdd-bridge agent runs by default (subject to `atdd.enabled`).
  - AC-002-02: Given `atdd.enabled: false` in `.isdlc/config.json`, When the discover orchestrator evaluates sub-phase 1d, Then the atdd-bridge is skipped.

- **FR-003** Introduce an `atdd` config section in `.isdlc/config.json` with four knobs, exposed through `ConfigService.getAtdd()`.
  - **Confidence**: High
  - Knobs: `enabled`, `require_gwt`, `track_red_green`, `enforce_priority_order`. All defaults `true`.
  - AC-003-01: Given `.isdlc/config.json` has no `atdd` section, When any hook or agent calls `ConfigService.getAtdd()`, Then the returned object equals `{ enabled: true, require_gwt: true, track_red_green: true, enforce_priority_order: true }`.
  - AC-003-02: Given `.isdlc/config.json` has a partial `atdd` section (e.g., only `require_gwt: false`), When `getAtdd()` is called, Then missing fields are filled with defaults and the partial override is preserved.

- **FR-004** Replace declarative `atdd_mode` conditionals in config files with runtime `ConfigService` checks in hooks, phase agents, and the phase-loop controller.
  - **Confidence**: High
  - AC-004-01: Given the refactor is complete, When `grep _when_atdd_mode src/isdlc/config/workflows.json` is run, Then zero matches are returned.
  - AC-004-02: Given the refactor is complete, When `grep '"when": "atdd_mode"' src/isdlc/config/iteration-requirements.json` is run, Then zero matches are returned.
  - AC-004-03: Given any Phase 05 or Phase 06 delegation, When the phase-loop controller builds the agent prompt, Then it injects `atdd.*` config values via GATE REQUIREMENTS INJECTION.

- **FR-005** When `require_gwt: true`, Phase 05 hard-blocks if Phase 01 produced acceptance criteria without Given/When/Then structure.
  - **Confidence**: High
  - AC-005-01: Given `atdd.require_gwt: true` and `requirements-spec.md` contains an AC lacking GWT structure, When `atdd-completeness-validator` runs at Phase 05 entry, Then it blocks with the non-GWT AC identified and an actionable error.
  - AC-005-02: Given `atdd.require_gwt: false` and a non-GWT AC is present, When Phase 05 runs, Then a best-effort scaffold is generated and flagged as `non_gwt: true` in atdd-checklist.json.

- **FR-006** When `track_red_green: true`, the test-watcher hook records RED→GREEN state transitions in atdd-checklist.json. When false, transition tracking is skipped.
  - **Confidence**: High
  - AC-006-01: Given `atdd.track_red_green: true` and a test transitions from failing to passing, When test-watcher observes the transition, Then an entry is appended to atdd-checklist.json with the transition timestamp and test id.
  - AC-006-02: Given `atdd.track_red_green: false`, When test-watcher observes any transition, Then no entry is written.

- **FR-007** When `enforce_priority_order: true`, Phase 06 requires tests to complete in priority order (P0 → P1 → P2 → P3). When false, any order is accepted.
  - **Confidence**: High
  - AC-007-01: Given `atdd.enforce_priority_order: true` and a P1 test passes while a P0 test still fails, When checkpoint-router evaluates progress, Then it blocks with an out-of-order error.
  - AC-007-02: Given `atdd.enforce_priority_order: false`, When tests pass in any order, Then checkpoint-router accepts the state.

- **FR-008** When `atdd.enabled: false`, all ATDD sub-behaviors become no-ops regardless of sub-knob values.
  - **Confidence**: High
  - AC-008-01: Given `atdd.enabled: false`, When Phase 05 runs, Then no atdd-checklist.json file is created and no ATDD scaffolds are generated.
  - AC-008-02: Given `atdd.enabled: false`, When the discover orchestrator reaches sub-phase 1d, Then atdd-bridge is skipped.
  - AC-008-03: Given `atdd.enabled: false`, When Phase 06 runs, Then RED→GREEN tracking and priority-order enforcement are skipped regardless of their sub-knob values.

- **FR-009** Update CLAUDE.md, docs/ARCHITECTURE.md, docs/HOOKS.md, and docs/AGENTS.md to reflect unconditional-by-default ATDD and the new config surface.
  - **Confidence**: High
  - AC-009-01: Given the refactor is complete, When docs are read, Then all references describe ATDD as default-on with config-file escape hatches (not as an opt-in flag).
  - AC-009-02: Given the refactor is complete, When the docs are grepped for `--atdd` or `--atdd-ready`, Then zero matches remain (except historical changelog entries).

## Assumptions and Inferences

- `ConfigService.getAtdd()` follows the accessor pattern established by GH-231 (e.g., `getMemory()`, `getBranches()`); adding the method is non-breaking to existing callers. **Confidence**: High (verified by Alex during codebase scan).
- `atdd-checklist.json` lives per-feature in `docs/requirements/{slug}/`, created by Phase 05 test-design-engineer and updated by Phase 06 software-developer. On re-runs (supervised redo, blast-radius re-implementation), the file is overwritten. **Confidence**: Medium (inherits existing artifact lifecycle; not explicitly verified).
- When `atdd.enabled: false`, sub-knob values are logically irrelevant (precedence rule). **Confidence**: High (design decision made during roundtable).
- Codex provider has no atdd-specific files (verified: grep on `src/providers/codex/**` returns zero matches), so dual-provider wiring reduces to parity verification. **Confidence**: High.
- `require_gwt: true` semantics (hard-block on non-GWT ACs) match the knob name; advisory behavior would weaken the contract. **Confidence**: High (design decision ADR-002).
- Test-file updates are covered within each hook/module implementation task per Article II (TDD); separate test-only tasks are not enumerated except where new files are introduced (ConfigService.getAtdd unit tests). **Confidence**: High.

## Non-Functional Requirements

| Attribute | Priority | Threshold |
|-----------|----------|-----------|
| Performance | Medium | Config read via ConfigService must complete in <5ms (file-cached, synchronous) |
| Testability | High | 4 knobs × {true,false} = 16 states; core states covered by unit tests, knob interactions covered by integration tests |
| Maintainability | High | Config centralization eliminates drift between workflows.json, iteration-requirements.json, and hook runtime guards |
| Compatibility | Low | Framework is unreleased; no migration path required |
| Reliability | High | ConfigService errors must fail-open to defaults (Article X) |

## Out of Scope

| Item | Reason |
|------|--------|
| Changing the ATDD protocol itself (skills, scaffold-generation logic, checklist schema) | Refactor scope is flag removal + config surface; behavioral logic unchanged |
| Per-phase config granularity (e.g., phase-05-atdd, phase-06-atdd) | User confirmed 4 global knobs is the right granularity |
| Deprecation warnings for stale `--atdd` / `--atdd-ready` flags | Framework is unreleased; no users to protect |
| Adding new ATDD behaviors beyond what currently exists | Refactor, not feature addition |
| Migration tooling for pre-refactor workflows | Framework is unreleased |
| Strict JSON schema validation of the `atdd` config section | Trust-the-user config model per existing convention (GH-231) |
